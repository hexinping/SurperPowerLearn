## Skill 速查表

| Skill | 触发场景 | 类型 |
|-------|---------|------|
| `brainstorming` | 新功能、新项目、任何创意工作 | 柔性 |
| `writing-plans` | 有设计规格，需要实施计划 | 柔性 |
| `subagent-driven-development` | 执行计划（推荐方式） | 柔性 |
| `executing-plans` | 在独立会话中执行计划 | 柔性 |
| `test-driven-development` | 任何功能实现或 bug 修复 | **刚性** |
| `systematic-debugging` | 任何 bug、测试失败、意外行为 | **刚性** |
| `verification-before-completion` | 声称任何工作完成之前 | **刚性** |

| `dispatching-parallel-agents` | 3+ 个独立问题需同时处理 | 柔性 |

| `finishing-a-development-branch` | 开发完成，需要收尾 | 柔性 |

## 三条铁律

```
1. 没有失败的测试，就没有生产代码     (TDD)
2. 没有根因调查，就没有修复           (Debugging)
3. 没有验证证据，就没有完成声明        (Verification)
```

---

## 标准开发流程

```
brainstorming → worktree → writing-plans → subagent-driven-dev → verification → finish-branch
    设计          环境          计划             实施               验证           收尾
```

---

## TDD 循环速记

```
RED    →  写失败测试  →  运行确认失败
GREEN  →  写最少代码  →  运行确认通过
REFACTOR → 清理重构  →  确认仍然通过
重复...
```

---

## 调试四阶段速记

```
Phase 1: 调查  →  读错误、复现、查变更、收集证据
Phase 2: 分析  →  找工作示例、对比差异
Phase 3: 假设  →  单一假设、最小测试、一次一个变量
Phase 4: 修复  →  写失败测试、单一修复、验证
         如果 3 次修复都失败 → 质疑架构！
```

---

## 验证门禁速记

```
IDENTIFY → RUN → READ → VERIFY → CLAIM
  什么命令    执行    读输出   确认结果   才能声明
```

---

## 子代理流程速记

```
每个任务:
  实现者 → 规格审查 → 代码审查
  (如果审查不过 → 修复 → 重新审查)

所有任务完成:
  最终审查 → finish-branch
```

---

## 模型选择速记

```
简单实现 (1-2 文件)  → haiku  (快、便宜)
集成任务 (多文件)    → sonnet (平衡)
架构/审查           → opus   (最强)
```

---

## 完成分支四选项

```
1. 本地合并    → merge + 删除分支 + 清理 worktree
2. 创建 PR    → push + gh pr create + 保留 worktree
3. 保留       → 什么都不做
4. 丢弃       → 需要确认 + 删除一切
```

## 危险信号速查

### 写代码时

```
❌ "太简单了不需要测试"         → 需要
❌ "我之后补测试"              → 现在写
❌ "先写代码再说"              → 先写测试
❌ "这个不需要设计"            → 需要（可以简短）
```

### 调试时

```
❌ "先试着改一下看看"           → 先调查根因
❌ "多改几个地方一起测"         → 一次一个变量
❌ "第 4 次修复应该行了"        → 停，质疑架构
❌ "我觉得是这个问题"           → 用证据证明
```

### 完成时

```
❌ "应该没问题了"              → 运行验证
❌ "测试之前通过过"            → 重新运行
❌ "代码改了就行了"            → 验证结果
❌ "太好了！完成了！"           → 先验证再说
```

## 计划文档禁止出现

```
❌ TBD / TODO / 稍后实现
❌ "添加适当的错误处理"
❌ "为上述编写测试"（不附代码）
❌ "与 Task N 类似"
❌ 只有描述没有代码的步骤
```
