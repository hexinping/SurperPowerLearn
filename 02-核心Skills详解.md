# 核心 Skills 详解

本文档详细介绍 SuperPowers 的 14 个 Skill，包括功能说明、工作流程、关键规则和使用示例。

---

## 1. brainstorming（头脑风暴）

### 定位
**所有创意工作的起点。** 在任何功能开发、组件构建、行为修改之前，必须先通过头脑风暴完成设计。

### 硬性门禁
> 在展示设计并获得用户批准之前，不得调用任何实现类 Skill、不得写任何代码、不得搭建任何项目。

### 工作流程

```
1. 探索项目上下文    → 检查文件、文档、最近的提交
2. 提供可视化伴侣    → 如果涉及视觉问题，询问是否需要浏览器展示
3. 逐个提问         → 一次一个问题，理解目的/约束/成功标准
4. 提出 2-3 种方案   → 带权衡分析和推荐
5. 展示设计         → 分段展示，每段获得批准
6. 编写设计文档     → 保存到 docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
7. 规格自审         → 检查占位符、矛盾、模糊性、范围
8. 用户审核规格     → 等待用户批准
9. 过渡到实施       → 调用 writing-plans skill
```

### 关键原则

- **一次一个问题** — 不要同时抛出多个问题
- **多选题优先** — 比开放式问题更容易回答
- **YAGNI** — 无情地删除不必要的功能
- **每个项目都需要设计** — 即使看起来很简单

### 反模式

```
❌ "这个太简单了，不需要设计"
   → 每个项目都走这个流程。简单项目的设计可以很短（几句话），但必须展示并获得批准。

❌ 在一条消息中同时提供可视化伴侣选项和问题
   → 可视化伴侣的提议必须是独立的消息。

❌ 设计完后直接开始写代码
   → 设计完后唯一的下一步是调用 writing-plans。
```

### 设计文档模板

```markdown
# [功能名称] 设计规格

## 概述
[1-2 句话描述要构建什么]

## 架构
[方案描述、组件关系]

## 数据流
[数据如何在系统中流动]

## 错误处理
[异常情况及处理方式]

## 测试策略
[如何验证功能正确]
```

## 2. writing-plans（编写计划）

### 定位
将设计规格转化为**详细的分步实施计划**。计划的读者假定为"零上下文的工程师"。

### 核心要求

- 每个步骤 **2-5 分钟** 可完成
- 包含**完整代码**（不是伪代码或占位符）
- 包含**精确的文件路径**
- 包含**验证步骤**和**预期输出**
- 遵循 **RED-GREEN-REFACTOR** TDD 循环

### 计划文档结构

```markdown
# [功能名称] 实施计划

> **For agentic workers:** 使用 superpowers:subagent-driven-development
> 或 superpowers:executing-plans 来逐任务执行此计划。

**Goal:** [一句话描述目标]
**Architecture:** [2-3 句话描述方案]
**Tech Stack:** [关键技术/库]

---

### Task 1: [组件名称]

**Files:**
- Create: `exact/path/to/file.py`
- Modify: `exact/path/to/existing.py:123-145`
- Test: `tests/exact/path/to/test.py`

- [ ] **Step 1: 编写失败的测试**
  [完整测试代码]

- [ ] **Step 2: 运行测试确认失败**
  Run: `pytest tests/path/test.py::test_name -v`
  Expected: FAIL

- [ ] **Step 3: 编写最小实现**
  [完整实现代码]

- [ ] **Step 4: 运行测试确认通过**
  Run: `pytest tests/path/test.py::test_name -v`
  Expected: PASS

- [ ] **Step 5: 提交**
  `git commit -m "feat: add specific feature"`
```

### 绝对禁止的占位符

以下内容在计划中**永远不应出现**：

```
❌ "TBD"、"TODO"、"稍后实现"
❌ "添加适当的错误处理"
❌ "为上述内容编写测试"（不附实际测试代码）
❌ "与 Task N 类似"（必须重复代码）
❌ 描述做什么但不展示怎么做的步骤
```

### 执行交接

计划完成后，提供两个选择：

1. **子代理驱动（推荐）** — 每个任务一个新子代理，任务间有审查
2. **内联执行** — 在当前会话中执行，批量执行带检查点

## 3. subagent-driven-development（子代理驱动开发）

### 定位
通过**独立子代理**逐任务执行计划。每个任务由新的子代理完成，保持上下文隔离。

### 核心流程

```
对于计划中的每个任务：
  1. 调度实现者子代理（提供任务完整文本 + 上下文）
  2. 实现者完成编码、测试、提交、自审
  3. 调度规格合规审查子代理
     └─ 通过？继续 / 不通过？实现者修复，重新审查
  4. 调度代码质量审查子代理
     └─ 通过？继续 / 不通过？实现者修复，重新审查
  5. 标记任务完成

所有任务完成后：
  6. 调度最终代码审查子代理
  7. 使用 finishing-a-development-branch skill
```

### 两阶段审查

| 阶段 | 审查者 | 关注点 |
|------|--------|--------|
| 第一阶段 | 规格合规审查 | 代码是否符合计划规格？有没有多做/少做？ |
| 第二阶段 | 代码质量审查 | 代码质量、架构、命名、可维护性 |

**重要顺序：** 必须先通过规格合规审查，才能进行代码质量审查。

### 实现者状态处理

| 状态 | 含义 | 处理方式 |
|------|------|----------|
| `DONE` | 完成 | 进入审查 |
| `DONE_WITH_CONCERNS` | 完成但有疑虑 | 先评估疑虑再审查 |
| `NEEDS_CONTEXT` | 需要更多信息 | 提供信息后重新调度 |
| `BLOCKED` | 受阻 | 评估原因并采取对策 |

### 模型选择策略

```
简单实现任务（1-2 文件，清晰规格） → 快速便宜模型 (haiku)
集成任务（多文件协调）              → 标准模型 (sonnet)
架构/设计/审查任务                  → 最强模型 (opus)
```

### 绝对禁止

```
❌ 跳过审查（规格合规或代码质量都不行）
❌ 审查有问题但继续下一个任务
❌ 并行调度多个实现子代理（会冲突）
❌ 让子代理自己读取计划文件（应提供完整文本）
❌ 先做代码质量审查再做规格合规审查
```

## 4. executing-plans（执行计划）

### 定位
在**独立会话**中执行计划，适合在并行会话中工作。

### 与 subagent-driven-development 的区别

| 特性 | subagent-driven | executing-plans |
|------|----------------|-----------------|
| 执行环境 | 当前会话 | 独立会话 |
| 上下文管理 | 每任务新子代理 | 批次执行带检查点 |
| 审查方式 | 两阶段自动审查 | 检查点处人工审查 |
| 适用场景 | 任务独立 | 需要并行会话 |

## 5. test-driven-development（测试驱动开发）

### 定位
强制执行 **RED-GREEN-REFACTOR** 循环。这是一个**刚性 Skill**，规则不可违反。

### 铁律

```
没有失败的测试，就没有生产代码。
先写代码后补测试？删除代码，从头开始。
```

### RED-GREEN-REFACTOR 循环

```
RED（红色）
  ├─ 编写一个最小的失败测试
  ├─ 运行测试，确认失败
  └─ 确认失败原因是"功能缺失"而非"语法错误"

GREEN（绿色）
  ├─ 编写最少的代码让测试通过
  ├─ 运行测试，确认通过
  └─ 确认其他测试也仍然通过

REFACTOR（重构）
  ├─ 移除重复
  ├─ 改善命名
  └─ 提取辅助函数（保持测试通过）
```

### 好测试 vs 坏测试

```python
# ✅ 好测试
def test_retries_failed_operations_3_times():
    """清晰的名称，测试真实行为，只测一件事"""
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception("fail")
        return "success"

    result = retry_operation(operation)
    assert result == "success"
    assert attempts == 3

# ❌ 坏测试
def test_retry_works():
    """模糊的名称，测试 mock 而非代码"""
    mock = MagicMock(side_effect=[Exception(), Exception(), "success"])
    retry_operation(mock)
    assert mock.call_count == 3
```

### 常见借口 vs 现实

| 借口 | 现实 |
|------|------|
| "太简单了不需要测试" | 简单的代码也会出错。测试只需 30 秒。 |
| "我之后补测试" | 之后写的测试立即通过，证明不了什么。 |
| "删除 X 小时的工作太浪费了" | 沉没成本谬误。保留不可信的代码才是浪费。 |
| "TDD 太教条了" | TDD 就是务实：比调试更快发现 bug。 |
| "先探索一下再说" | 可以探索。但探索完必须丢掉代码，用 TDD 重写。 |

### 完成检查清单

```
□ 每个新函数/方法都有测试
□ 看到了每个测试失败
□ 每个测试因正确原因失败（功能缺失，非拼写错误）
□ 写了最少的代码让测试通过
□ 所有测试通过
□ 输出干净（无错误、无警告）
□ 测试使用真实代码（只在必要时使用 mock）
□ 覆盖了边界情况和错误场景
```

## 6. systematic-debugging（系统化调试）

### 定位
遇到**任何 bug、测试失败或意外行为**时使用。这是一个**刚性 Skill**。

### 铁律

```
没有根因调查，就没有修复。
```

### 四阶段流程

#### 第一阶段：根因调查

```
1. 仔细阅读错误信息  → 不要跳过，完整阅读堆栈跟踪
2. 稳定复现          → 能可靠触发吗？具体步骤？
3. 检查最近变更      → git diff、最近提交、新依赖
4. 多组件系统中收集证据 → 在每个组件边界添加诊断日志
5. 追踪数据流        → 坏值从哪里来？一路向上追溯
```

#### 第二阶段：模式分析

```
1. 找到工作的示例    → 代码库中相似但正常工作的代码
2. 与参考比较        → 完整阅读参考实现（不要略读）
3. 识别差异          → 列出工作和损坏之间的每个差异
4. 理解依赖          → 需要什么组件/配置/环境？
```

#### 第三阶段：假设与测试

```
1. 形成单一假设      → "我认为 X 是根因，因为 Y"
2. 最小测试          → 做最小的改变来测试假设
3. 一次一个变量      → 不要同时修改多处
4. 验证              → 有效？进入第四阶段 / 无效？新假设
```

#### 第四阶段：实施修复

```
1. 创建失败的测试用例  → 用测试复现 bug
2. 实施单一修复        → 只修根因，不做"顺便"的改进
3. 验证修复            → 测试通过？其他测试没破？
4. 如果 3 次修复都失败  → 停下来，质疑架构
```

### 危险信号

```
❌ "先快速改一下看看行不行"
❌ "先试着改 X"
❌ "多改几个地方一起跑测试"
❌ "我觉得是 X，直接修了"
❌ "不完全理解但这样应该行"
```

看到这些想法？**停下来，回到第一阶段。**

## 7. verification-before-completion（完成前验证）

### 定位
在**声称任何工作完成之前**必须执行。这是一个**刚性 Skill**。

### 铁律

```
没有新鲜的验证证据，就没有完成声明。
```

### 门禁函数

```
在声称任何状态或表达满意之前：

1. IDENTIFY — 什么命令能证明这个声明？
2. RUN     — 执行完整命令（新鲜的、完整的）
3. READ    — 完整输出，检查退出码，统计失败数
4. VERIFY  — 输出是否确认了声明？
5. CLAIM   — 只有在确认后才能声明

跳过任何步骤 = 说谎，不是验证
```

### 常见失败

| 声明 | 需要 | 不够的 |
|------|------|--------|
| "测试通过" | 测试命令输出：0 失败 | 之前的运行、"应该通过" |
| "构建成功" | 构建命令：exit 0 | lint 通过、日志看着没问题 |
| "Bug 修好了" | 测试原始症状：通过 | 代码改了就假定修好了 |
| "需求满足了" | 逐行检查清单 | 测试通过 |

### 危险信号

```
❌ 使用"应该"、"可能"、"看起来"
❌ 在验证前表达满意（"太好了！"、"完成！"）
❌ 准备提交/PR 但没验证
❌ 信任代理的成功报告
```

## 8. using-git-worktrees（Git Worktree 使用）

### 定位
为开发工作创建**隔离的 Git 工作树**，避免影响主分支。

### 工作流程

```
1. 创建 worktree     → 在 .worktrees/ 或全局位置
2. 验证 .gitignore   → 确保 worktree 目录被忽略
3. 自动检测项目名称   → 基于目录名
4. 创建特性分支      → 基于当前 HEAD
5. 在 worktree 中工作
6. 完成后清理        → 使用 finishing-a-development-branch
```

## 9. requesting-code-review（请求代码审查）

### 定位
使用 `superpowers:code-reviewer` 代理对已完成的工作进行专业审查。

### 审查维度

| 维度 | 关注点 |
|------|--------|
| 计划对齐 | 代码是否符合原始计划规格？ |
| 代码质量 | 命名、结构、可读性 |
| 架构设计 | 模式选择、组件边界 |
| 文档标准 | 必要的注释和文档 |
| 问题识别 | 潜在 bug、安全问题 |

---

## 10. receiving-code-review（接收审查反馈）

### 定位
正确处理审查反馈。

### 关键原则

```
1. 技术评估 — 评估建议的技术价值，不是情感反应
2. 验证建议 — 在实施审查建议前先验证其正确性
3. 逐条处理 — 不要批量处理审查意见
```

## 11. dispatching-parallel-agents（并行代理调度）

### 定位
当有 **3 个以上独立问题**需要同时处理时使用。

### 使用条件

```
✅ 3+ 个独立的测试失败
✅ 问题之间没有依赖关系
✅ 每个问题可以独立修复

❌ 问题之间有依赖
❌ 需要协调的任务
❌ 少于 3 个问题
```

### 工作方式

```
1. 按问题域分组
2. 为每个组调度独立代理
3. 各代理并行工作
4. 收集结果
```

## 12. finishing-a-development-branch（完成开发分支）

### 定位
开发完成后，引导选择如何集成工作。

### 流程

```
Step 1: 验证测试通过   → 测试不通过则停止
Step 2: 确定基准分支   → 通常是 main 或 master
Step 3: 提供 4 个选项  → 合并/PR/保留/丢弃
Step 4: 执行选择
Step 5: 清理 worktree
```

### 四个选项

| 选项 | 合并 | 推送 | 保留Worktree | 清理分支 |
|------|------|------|-------------|----------|
| 1. 本地合并 | ✓ | - | - | ✓ |
| 2. 创建 PR | - | ✓ | ✓ | - |
| 3. 保留 | - | - | ✓ | - |
| 4. 丢弃 | - | - | - | ✓ (强制) |

## 13. writing-skills（编写 Skill）

### 定位
创建新的自定义 Skill。将 TDD 的思想应用于流程文档。

### 方法

```
1. 将 Skill 编写视为对流程的 TDD
2. 定义 Skill 的目的和规则
3. 用子代理进行压力测试
4. 迭代改进
```

## 14. using-superpowers（使用 SuperPowers）

### 定位
系统入口 Skill。定义了如何发现和使用其他 Skill。

### 核心规则

```
即使只有 1% 的可能性某个 Skill 适用，也必须调用它。
```

### Skill 优先级

```
1. 流程 Skill 优先（brainstorming、debugging）→ 决定如何接近任务
2. 实现 Skill 其次 → 指导执行
```

### 危险思维

| 你的想法 | 现实 |
|----------|------|
| "这只是一个简单问题" | 问题也是任务，检查 Skill |
| "我先了解一下再说" | Skill 告诉你如何了解 |
| "这不需要正式的 Skill" | 如果 Skill 存在，就使用它 |
| "我记得这个 Skill" | Skill 会更新，读当前版本 |
