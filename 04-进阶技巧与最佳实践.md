## 1. 子代理（Subagent）深入理解

### 1.1 什么是子代理？

子代理是 Claude Code 中的**独立 AI 工作进程**。每个子代理：

- 拥有独立的上下文窗口（不共享主会话的上下文）
- 只接收到你明确提供的信息
- 完成任务后将结果返回给主会话
- 可以使用不同的模型（按复杂度选择）

```
主会话 (Orchestrator)
  │
  ├─ 子代理 A (Implementer) — 实现任务 1
  ├─ 子代理 B (Spec Reviewer) — 审查任务 1 的规格合规性
  ├─ 子代理 C (Quality Reviewer) — 审查任务 1 的代码质量
  │
  ├─ 子代理 D (Implementer) — 实现任务 2
  ├─ 子代理 E (Spec Reviewer) — 审查任务 2
  ...
```

### 1.2 子代理上下文管理

**好的做法：** 在调度子代理时提供完整信息

```
✅ "实现 Task 3。以下是完整的任务描述：[完整文本]。
    以下是你需要知道的项目上下文：[相关信息]。
    以下是已完成的 Task 1 和 Task 2 的接口定义：[接口]。"
```

**坏的做法：** 让子代理自己找信息

```
❌ "去读 docs/plan.md 文件中的 Task 3 并实现它。"
    → 子代理浪费时间读文件、理解上下文
    → 可能误解计划意图
```

### 1.3 模型选择策略详解

SuperPowers 推荐根据任务复杂度选择模型：

| 任务类型 | 推荐模型 | 选择理由 |
|---------|---------|---------|
| 单文件实现，规格清晰 | haiku | 速度快、成本低，足够处理 |
| 多文件协调，需要理解上下文 | sonnet | 平衡能力和成本 |
| 架构设计、复杂审查 | opus | 需要最强推理能力 |
| 简单的规格合规审查 | haiku/sonnet | 对照检查，不需要深度推理 |
| 代码质量审查 | sonnet/opus | 需要经验判断 |

**实际使用示例：**

```python
# 在 Claude Code 中使用 Agent 工具时指定模型
# 简单实现任务
Agent(model="haiku", prompt="实现 Task 1...")

# 复杂架构任务
Agent(model="opus", prompt="审查整体架构...")
```

### 1.4 处理子代理失败

当子代理报告 `BLOCKED` 状态时的升级策略：

```
失败分析：
  │
  ├─ 缺少上下文？
  │  └─ 提供更多上下文，重新调度（同模型）
  │
  ├─ 任务太复杂？
  │  └─ 用更强的模型重新调度
  │
  ├─ 任务太大？
  │  └─ 拆分成更小的子任务
  │
  └─ 计划本身有问题？
     └─ 升级给用户，修改计划
```

## 2. Git Worktree 高级用法

### 2.1 Worktree 的价值

```
没有 Worktree:
  main ─── 你的修改（如果出错，影响主分支）

有 Worktree:
  main ─── 不受影响
  .worktrees/feature-x ─── 你的修改（隔离的安全空间）
```

### 2.2 多 Worktree 并行开发

```
项目根目录/
├── .worktrees/
│   ├── feature-auth/     ← Agent A 在这里工作
│   ├── feature-api/      ← Agent B 在这里工作
│   └── bugfix-db/        ← Agent C 在这里工作
├── .gitignore            ← 包含 .worktrees/
└── src/                  ← 主分支不受影响
```

### 2.3 Worktree 生命周期

```
创建 → 开发 → 验证 → 合并/丢弃 → 清理

重要：完成后一定要清理，避免磁盘空间浪费
```

## 3. 设计文档最佳实践

### 3.1 规格文档的关键要素

一份好的规格文档应包含：

```markdown
# [功能] 设计规格

## 1. 概述（1-2 句话）
做什么、为什么做

## 2. 架构（图 + 文字）
组件如何交互

## 3. 数据模型
输入/输出/存储的数据结构

## 4. 接口定义
API 端点、函数签名、参数

## 5. 错误处理
可能出错的情况及处理方式

## 6. 测试策略
如何验证功能正确

## 7. 约束和假设
性能要求、依赖假设、范围限制
```

### 3.2 规格自审检查清单

写完规格后，执行以下检查：

```
□ 占位符扫描：有没有 "TBD"、"TODO"？
□ 内部一致性：各部分是否矛盾？
□ 范围检查：是否足够聚焦？是否需要拆分？
□ 歧义检查：有没有可以被两种方式理解的需求？
```

## 4. TDD 进阶技巧

### 4.1 测试金字塔

```
        /  E2E  \        ← 少量，覆盖关键路径
       / 集成测试 \       ← 中等，验证组件交互
      /  单元测试   \     ← 大量，验证每个单元
     ──────────────────
```

### 4.2 何时使用 Mock

```
✅ 使用 Mock 的场景：
   - 外部 API 调用（网络不稳定）
   - 数据库操作（使用内存数据库替代）
   - 文件系统操作（使用临时目录）
   - 时间相关操作（固定时间点）

❌ 不应使用 Mock 的场景：
   - 测试自己的代码逻辑
   - 测试简单的数据转换
   - Mock 链超过 2 层深度
```

### 4.3 测试命名规范

```python
# ✅ 好的命名 — 描述行为
def test_empty_email_returns_validation_error():
def test_expired_token_is_rejected():
def test_admin_can_delete_any_user():

# ❌ 差的命名 — 模糊或描述实现
def test_email():
def test_token():
def test_delete():
```

### 4.4 处理遗留代码

```
遗留代码（没有测试的代码）怎么办？

1. 不要试图一次性补全所有测试
2. 对你要修改的部分添加测试
3. 先写一个"表征测试"（描述当前行为）
4. 然后用 TDD 添加新行为
5. 逐步提高覆盖率
```

## 5. 调试进阶

### 5.1 多组件系统的诊断

当 bug 跨越多个组件时：

```
方法：在每个组件边界添加日志

前端 → API 网关 → 业务服务 → 数据库
  │        │          │          │
  └ log    └ log      └ log      └ log

通过日志确定问题在哪两个组件之间
```

### 5.2 根因追踪技术

```
从症状向上追溯：

错误: "NoneType has no attribute 'name'"
  ↑
user 变量为 None
  ↑
get_user(id) 返回了 None
  ↑
数据库中没有这个 ID 的用户
  ↑
前端传递了错误的用户 ID ← 根因！
```

### 5.3 三次失败规则

```
修复尝试 #1 失败？ → 重新分析，尝试修复 #2
修复尝试 #2 失败？ → 深入调查，尝试修复 #3
修复尝试 #3 失败？ → 停！这不是 bug，这是架构问题

3 次修复失败意味着：
  - 你在修症状，不是修根因
  - 架构本身有问题
  - 需要重新设计，而不是继续打补丁
```

## 6. 代码审查最佳实践

### 6.1 两阶段审查的价值

```
为什么要分两阶段？

阶段 1（规格合规）回答：做对了吗？
  - 代码做了计划要求的事吗？
  - 有没有少做或多做？

阶段 2（代码质量）回答：做好了吗？
  - 代码可读吗？
  - 架构合理吗？
  - 有没有性能问题？

先确认"做对了"，再讨论"做好了"。
顺序不能反，因为"做好了但做错了"毫无意义。
```

### 6.2 审查反馈的处理

```
收到审查反馈后：

1. 不要情绪化 — 这是技术评估，不是人身攻击
2. 逐条评估 — 每条建议都独立考虑
3. 验证建议 — 在实施前确认建议的正确性
4. 不要批量处理 — 一次处理一条反馈
5. 修复后重新审查 — 不要跳过重新审查步骤
```

## 7. 自定义 Skill 开发

### 7.1 什么时候需要自定义 Skill？

```
✅ 需要自定义 Skill：
   - 团队有特定的开发流程
   - 项目有特殊的架构约束
   - 反复出现相同的工作模式

❌ 不需要自定义 Skill：
   - 一次性的任务
   - 已有 Skill 能覆盖的场景
```

### 7.2 Skill 文件结构

```
skills/
  my-custom-skill/
    SKILL.md          ← Skill 定义（必需）
    helper-file.md    ← 辅助文件（可选）
```

### 7.3 SKILL.md 模板

```markdown
---
name: my-custom-skill
description: 一句话描述，用于自动匹配
---

# Skill 名称

## Overview
做什么、为什么需要

## When to Use
触发条件

## The Process
具体步骤

## Red Flags
不应该做的事

## Integration
与其他 Skill 的关系
```

## 8. 性能与成本优化

### 8.1 减少 Token 消耗

```
策略 1: 精准的子代理上下文
  - 只提供任务需要的信息
  - 不要复制整个计划给每个子代理

策略 2: 合理的模型选择
  - 简单任务用便宜模型
  - 只在需要时用强力模型

策略 3: 避免不必要的审查循环
  - 计划写得越详细，审查返工越少
  - 规格越清晰，实现越准确
```

### 8.2 减少开发时间

```
策略 1: 好的设计减少返工
  brainstorming 阶段多花 10 分钟
  → 实施阶段少返工 1 小时

策略 2: 并行工作
  独立任务用 dispatching-parallel-agents
  → 3 个独立问题同时修复

策略 3: 避免调试循环
  systematic-debugging 一次找到根因
  → 避免 5 次猜测式修复
```

## 9. 常见问题与解决方案

### Q1: SuperPowers 让开发变慢了？

```
短期：是的，走流程比直接写代码慢
长期：不是，因为：
  - 设计阶段减少了返工
  - TDD 减少了调试时间
  - 审查减少了 bug
  - 系统化调试比猜测快

权衡：前期多花 20% 时间，后期少花 50% 时间调试
```

### Q2: 简单任务也要走完整流程？

```
不一定。SuperPowers 的规则是：
  - brainstorming：每个项目都要，但设计可以很简短
  - TDD：每个功能/bug 修复都要
  - verification：每次完成声明都要
  - 完整的子代理流程：只在复杂项目需要
```

### Q3: 可以跳过 TDD 吗？

```
官方立场：不行（刚性 Skill）
实际情况：可以在 CLAUDE.md 中覆盖

如果你在 CLAUDE.md 中写：
  "对于原型开发，不要求 TDD"
那么 SuperPowers 会遵从你的指令。

但 SuperPowers 强烈不推荐这样做。
```

### Q4: 子代理的审查太慢了？

```
v5.0.7 的优化：
  - 内联自审查取代了之前 25 分钟的子代理审查循环
  - 保持相同质量，大幅减少时间

你也可以：
  - 对简单任务跳过代码质量审查（只保留规格合规审查）
  - 对明显正确的实现简化审查流程
```

### Q5: 如何与团队协作？

```
SuperPowers 主要面向单人开发者 + AI 助手的工作模式。
但其产出物（设计文档、计划、PR）完全适合团队协作：

1. 设计文档 → 团队评审
2. 实施计划 → 任务分配
3. 代码审查 → 团队 review
4. PR → 标准 Git 工作流
```
