# 开发工作流实战

本文档通过**完整的端到端示例**，展示如何使用 SuperPowers 的各个 Skill 串联成完整的开发工作流。

---

## 1. 完整工作流总览

SuperPowers 的标准开发流程如下：

```
用户提出需求
    │
    ▼
┌─────────────────────┐
│  brainstorming      │  ← 头脑风暴，生成设计文档
│  (设计阶段)          │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  using-git-worktrees│  ← 创建隔离工作空间
│  (环境准备)          │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  writing-plans      │  ← 编写详细实施计划
│  (规划阶段)          │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  subagent-driven-   │  ← 逐任务执行（推荐）
│  development        │     或 executing-plans
│  (实施阶段)          │
│                     │
│  ┌───────────────┐  │
│  │ TDD 循环       │  │  ← 每个任务内部
│  │ RED → GREEN    │  │
│  │   → REFACTOR   │  │
│  └───────────────┘  │
│                     │
│  ┌───────────────┐  │
│  │ 规格合规审查    │  │  ← 每个任务完成后
│  │ 代码质量审查    │  │
│  └───────────────┘  │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  verification-      │  ← 最终验证
│  before-completion  │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│  finishing-a-       │  ← 合并/PR/保留/丢弃
│  development-branch │
└─────────────────────┘
```

## 2. 实战场景：开发一个待办事项 API

### 场景描述

> 用户："我想用 Python + FastAPI 做一个简单的待办事项 REST API，支持增删改查。"

### Step 1: 头脑风暴（brainstorming）

Claude 自动调用 `brainstorming` skill。

**探索上下文：**

```
Claude: 我正在使用 brainstorming skill 来设计这个功能。
        让我先了解一下当前项目状态...
        [检查项目文件、依赖、现有代码]
```

**逐个提问（一次一个）：**

```

  A) SQLite（轻量级，适合学习和小项目）(推荐)
  B) PostgreSQL（生产级数据库）
  C) 内存字典（最简单，重启丢失数据）

用户: A

Claude: API 需要认证吗？
  A) 不需要，先做基础功能 (推荐)
  B) 简单的 API Key
  C) JWT 认证

用户: A
```

**提出方案：**

```
Claude: 基于我们的讨论，我推荐以下方案：

方案 1（推荐）: FastAPI + SQLAlchemy + SQLite
  - 优势：类型安全、自动文档、ORM 简化操作
  - 劣势：多一层抽象

方案 2: FastAPI + 原生 SQLite
  - 优势：直接控制 SQL
  - 劣势：手写 SQL 容易出错

方案 3: Flask + SQLAlchemy
  - 优势：Flask 更简单
  - 劣势：缺少自动文档和类型检查

**展示设计（分段，每段确认）：**

## 3. 调试场景：修复一个 Bug

### 场景描述

> 用户："创建待办事项时，如果标题为空字符串，API 返回 500 而不是 400。"

### 使用 systematic-debugging

```
Claude: 我正在使用 systematic-debugging skill 来调查这个问题。
```

**第一阶段：根因调查**

```
1. 复现问题：
   $ curl -X POST http://localhost:8000/todos -d '{"title": ""}'
   → 500 Internal Server Error

2. 查看错误日志：
   sqlalchemy.exc.IntegrityError: NOT NULL constraint failed
   → 数据库层报错了，说明请求未经验证就到了数据库

3. 检查相关代码：
   schemas.py → Pydantic 模型没有对空字符串做验证
   crud.py → 直接接收并插入数据库
```

**第二阶段：模式分析**

```
找到类似的工作代码：update 端点有标题验证
比较差异：create 端点缺少 min_length 验证
```

**第三阶段：假设**

```
假设：schemas.py 中 TodoCreate 的 title 字段缺少 min_length=1 约束
```

**第四阶段：TDD 修复**

```
Claude: 使用 TDD 方式修复...

RED:
  def test_create_todo_empty_title_returns_422():
      response = client.post("/todos", json={"title": ""})
      assert response.status_code == 422

  $ pytest tests/test_api.py::test_create_todo_empty_title -v
  FAIL: assert 500 == 422  ✓ 按预期失败

GREEN:
  # schemas.py
  class TodoCreate(BaseModel):
      title: str = Field(min_length=1)  # 添加验证

  $ pytest tests/test_api.py -v
  7 passed  ✓

验证修复：
  $ curl -X POST http://localhost:8000/todos -d '{"title": ""}'
  → 422 Unprocessable Entity  ✓
```

## 4. 简化场景：快速任务

并非所有任务都需要走完整流程。对于简单任务，流程会自动简化：

### 简单 Bug 修复

```
用户: "修复首页的拼写错误"

流程:
  1. systematic-debugging → 定位错误位置
  2. test-driven-development → 如果可测试则 TDD
  3. verification-before-completion → 验证修复
  （跳过 brainstorming、writing-plans 等）
```

### 添加小功能

```
用户: "给 API 加一个健康检查端点 /health"

流程:
  1. brainstorming → 快速设计（几句话）
  2. test-driven-development → TDD 实现
  3. verification-before-completion → 验证
  （跳过 worktree、详细计划等）
```

## 6. 工作流决策树

```
你要做什么？
│
├─ 新功能/新项目
│  └─ brainstorming → writing-plans → subagent-driven-development
│
├─ 修复 Bug
│  └─ systematic-debugging → TDD 修复 → verification
│
├─ 重构代码
│  └─ brainstorming（如果影响大）→ TDD → verification
│
├─ 多个独立问题
│  └─ dispatching-parallel-agents
│
├─ 审查代码
│  └─ requesting-code-review → receiving-code-review
│
└─ 完成开发
   └─ verification → finishing-a-development-branch
```

## 7. Skill 之间的调用关系

```
brainstorming
  └─→ writing-plans
        └─→ subagent-driven-development (推荐)
        │     ├─→ test-driven-development (每个任务内)
        │     ├─→ requesting-code-review (审查)
        │     └─→ finishing-a-development-branch (完成)
        │
        └─→ executing-plans (替代方案)
              ├─→ test-driven-development
              └─→ finishing-a-development-branch

systematic-debugging
  ├─→ test-driven-development (修复阶段)
  └─→ verification-before-completion (验证修复)
```
